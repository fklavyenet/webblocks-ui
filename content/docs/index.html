Read from the inside out

The docs begin with structure, not promotion.

Start with foundations, then layout primitives, then UI primitives, then utilities and JavaScript. Pair that reading order with pattern-first composition so you can see where structure, visible building blocks, and behavior each actually come from.

What to expect

  • Foundations first

    Colors, spacing, radius, shadows, type, and theme axes are the base layer.

  • Layouts before flourishes

    Containers and shells determine how the interface breathes before individual widgets do.

  • Patterns first

    Start from auth, dashboard, form, and marketing patterns before drilling into individual primitives.

Reading order

A practical path through the system.

If you are new to the system, follow this order. It mirrors the build and the way a real page is assembled.

  1. Foundation

    Read the token system, theme axes, and the rules around accent, radius, border, density, and font.

  2. Patterns

    Pick a real page job first: auth, dashboard, form, or marketing.

  3. Layout primitives

    Choose the shell and flow tools: page, section, container, stack, row, grid, or shell layout.

  4. UI primitives

    Add the building blocks the page needs: forms, tables, alerts, overlays, lists, filters, navigation.

  5. Utilities and JS

    Only then reach for spacing helpers, visibility helpers, and behavioral modules.

Build model

The docs follow the build order because the build order matters.

WebBlocks is not assembled by an opaque toolchain. The final bundle is the result of an explicit file list. If a token or layout primitive appears earlier in the build, it is because later layers depend on it.
