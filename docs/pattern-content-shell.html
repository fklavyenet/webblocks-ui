Content shell example

Use the content shell when hierarchy, reading width, references, and calm progression matter more than application chrome.

Docs surfaceCentered reading widthPattern-first
When to use it Choose the content shell for docs, onboarding, policies, changelogs, and longer articles where page hierarchy should stay obvious.

Why content shells stay narrow

Documentation and article pages work best when the layout keeps reading width stable, separates header metadata from body content, and uses footer navigation for linear progression.

Need persistent navigation?Use dashboard shell
Operational products should switch to the dashboard frame instead of stretching a content shell into an app shell.
Need grouped form work?Use settings shell
Settings pages need local navigation and sectioned controls, not only article flow.

Reference blocks still use shipped primitives

The content shell does not invent its own building blocks. Cards, alerts, lists, and tables still come from the primitive layer.

NeedPrimitiveWhy
Inline warningwb-calloutKeeps emphasis inside the reading flow.
Cross-linkswb-link-listMakes adjacent docs or examples easy to scan.
Structured comparisonwb-tableClarifies differences without breaking the shell.
The shell sets the page job. Primitives still do the visible UI work inside it.
