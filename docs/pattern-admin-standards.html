Admin Standards

Use these patterns for admin create, edit, detail, and index/list screens. The standard keeps hierarchy outside cards, keeps filters and list cards consistent across products, keeps form submit actions inside form footers, and keeps destructive work isolated in a clear final danger zone.

Pattern guidanceCSS-firstCopyable markup
Standard goal Page title and breadcrumb explain where the user is. Cards and sections explain what they can do next. Submit actions belong to the form they submit, not to the global page header.
JobPreferred structureRule
Admin create or edit pagewb-page-header + wb-card + wb-form-sectionKeep the page h1 outside cards and group fields by purpose.
Detail pagewb-detail-list + supporting cardsUse label/value rows instead of loose paragraphs.
Index tablewb-table + wb-table-actionsAlways label the final column Actions and let the header and buttons start at the left edge of that cell.
Destructive actionwb-danger-zoneKeep it in the final card with a specific consequence label.
Preferred classesPurposeNotes
wb-page-header, wb-page-breadcrumb, wb-page-title, wb-page-description, wb-page-actionsPage-level admin hierarchyUse one page h1 outside cards and keep only context actions here. wb-page-breadcrumbs remains a compatibility alias.
wb-breadcrumb, wb-breadcrumb-item, wb-breadcrumb-link, wb-breadcrumb-currentSecondary hierarchy contextSeparators are CSS-owned by default. wb-breadcrumb-separator remains only as a manual escape hatch.
wb-card, wb-card-footer, wb-field, wb-field-hint, wb-field-error, wb-action-group, wb-grid-2, wb-stack-4Existing card, field, action, and layout structureUse existing primitives first. The v2.6.0 form aliases stay available for compatibility.
wb-form-section, wb-form-section-header, wb-form-section-title, wb-form-section-description, wb-form-error-summary, wb-input-monoAdmin-specific form semanticsKeep these when the markup needs explicit sectioning, whole-form validation summary, or code-like input treatment.
wb-detail-list, wb-detail-row, wb-detail-label, wb-detail-value, wb-detail-value-monoDetail page label/value rowsUse monospace values for paths, tokens, commands, and URLs.
td.wb-table-actionsExplicit final row action cellPair it with a normal left-aligned Actions table header and left-aligned action buttons. Keep wb-action-group generic outside table cells.
wb-danger-zone, wb-danger-zone-title, wb-danger-zone-description, wb-danger-zone-actionsFinal destructive sectionUse specific destructive labels such as Delete CMS Installation.

Admin index/list standard

Canonical contract Admin index/list screens in CMS, Publisher, QuizTem, Herne, and future projects should share this DOM shape so table spacing, filter placement, row actions, and pagination behave the same way.
AreaRule
Page headerKeep wb-page-header at page level. It owns the page title, breadcrumb, description, and only page-context actions.
FiltersRender filters before the list card as a separate compact wb-filter-bar or filter card. Do not place filters inside the table card body.
List cardRender the list in section.wb-card. The card header owns the list title, description, and right-side create/import/export actions.
Card spacingwb-card-body should contain wb-table-wrap directly. Avoid extra project-specific wrappers around the table.
TableUse wb-table wb-table-hover. Add wb-table-striped only when striping is intentionally needed.
Action columnNever leave the final header blank. Label it Actions, and put row controls in td.wb-table-actions.
Row actionsWrap action icons in wb-action-group and render each action as wb-action-btn.
Dense technical valuesRender product keys, slugs, source paths, checksums, versions, route names, and similar machine values as compact monospace/code values, for example <code class="wb-nowrap">webblocks-ui</code>, so the value remains scannable and the table can scroll inside wb-table-wrap instead of forcing aggressive character wrapping.
Long status summariesLet human-readable status and last-run summaries wrap as normal prose. Keep them concise in the table; move diagnostic detail to a detail page or secondary metadata block when it competes with identity, status, and actions.
PaginationPlace pagination or result summary in wb-card-footer, not above the table and not inside wb-table-wrap.
Legacy class guidance Do not use project-specific wrappers such as wb-admin-table-card, wb-admin-table-card-body, or wb-admin-pages-table-wrap in new canonical examples. If older projects need compatibility, they may keep CSS aliases that map those names to the canonical wb-card, wb-card-body, and wb-table-wrap behavior.

Admin Filter Bar Actions

Filter action standard Admin filter forms use Filter for the primary submit action and Clear for clearing active filters. Render Clear only when at least one filter is active.
AreaRule
Form structureUse a real form with wb-filter-bar. Put fields in wb-filter-bar-start and action buttons in wb-filter-bar-end.
Field and action alignmentUse existing composition such as wb-items-end, wb-field, wb-flex-1, and wb-min-w-0 so labels, controls, and action buttons share the same baseline and field rhythm.
LabelsThe canonical English labels are Filter and Clear. Do not use Apply, Apply Filters, Reset, or Clear filters on new WebBlocks admin filter surfaces.
Clear visibilityRender Clear only when filters are active. Omit it entirely for the unfiltered default state instead of disabling it.
Button sizeUse normal wb-btn buttons so filter actions match input and select height. Do not use wb-btn-sm for filter bar action buttons.
Clear variantUse an existing secondary/subtle WebBlocks button style such as wb-btn-ghost; do not invent a new clear-button variant.
ScopeThis rule is for filter bar actions. Row actions and compact toolbar icon buttons keep their own icon-button and compact-action standards.
CSSDo not write project-specific filter alignment CSS. Compose existing WebBlocks primitives and utilities instead of adding a dedicated alignment class.

Dense table content and technical values

Technical value rule Long machine-readable values in admin tables should stay visually distinct from human copy. Use compact monospace/code presentation for product keys, slugs, source paths, checksums, versions, route names, queue names, command names, and similar identifiers. Prefer existing WebBlocks UI primitives and utilities such as code, wb-mono, wb-text-sm, and wb-nowrap; do not add project-local table CSS just to bypass the shared wrapping behavior.

Content typeTable treatmentWhen to move it out
Product key, slug, source path, checksum, version, routeUse compact monospace/code styling. Add wb-nowrap when preserving the complete token is more readable than character-level wrapping and the table is already inside wb-table-wrap.Move full values to a detail page or secondary metadata block when several long technical values make the row harder to scan.
Last status or last-run summaryUse normal prose wrapping. Keep it short enough for the row to remain readable next to identity and actions.Move logs, validation lists, stack traces, or multi-step diagnostics to a detail page, modal, or secondary metadata block.
ActionsKeep the final header as Actions. Use td.wb-table-actions and a single wb-action-group so the column remains narrow, stable, and aligned.Use a detail page when a row needs many operations or explanatory action copy.
Confirmation flowsUse wb-modal and the shared overlay root for destructive, import, publish, deploy, or re-run confirmation.Do not use browser confirm(); if the flow needs more context, put the context in the modal body.

Health and notices

Optional notices should not automatically read as infrastructure failures.

SiteStatusSummaryActions

Action column rule Never leave an action-only header blank. Use the explicit Actions label, keep the header left-aligned like the rest of the table, and put row controls in td.wb-table-actions. The generic wb-action-group may wrap outside table cells, while grouped actions inside a table action cell stay on one line.

Health and notice status guidance

Do

Make the primary system state explicit, then describe secondary notices clearly.

Use labels such as Healthy with notices when required checks pass.

Let long health summaries wrap instead of clipping action-heavy rows.

Keep optional integrations visually secondary to required infrastructure state.

Avoid

Do not flatten every notice into a generic error state.

Do not mark the whole system degraded when only an optional integration is disabled.

Do not hide long summaries behind clipped one-line cells when diagnosis matters.

Do not use vague destructive labels such as just Delete.

Copy and hierarchy rules

Page titleUse one page h1 outside cards. Card titles should not replace it.
Card titlesUse h2 inside card headers. Use h3 only for nested sections inside a card.
Primary submit labelsUse Create {Resource} for create screens and Save Changes for edit screens.
Cancel orderingKeep the primary action first and the secondary or cancel action second in the same footer group.
Danger labelsUse specific labels such as Delete CMS Alias, Delete Site, or Delete User.
